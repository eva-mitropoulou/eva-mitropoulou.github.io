ScreenSift · virtual-screening triage

ScreenSift is a Python package built around a practical virtual-screening problem: how do we combine 2D ligand-based evidence and 3D structure-based evidence into one ranked ligand table while still preserving the reason each compound was selected?

The workflow starts from a ligand table, curates molecular structures with RDKit, prepares docking inputs with Meeko, screens compounds with Uni-Dock, rescores docked poses with GNINA, and optionally adds ECFP4/Tanimoto similarity against known actives. The final output is a transparent triage table that ranks compounds while preserving the evidence behind each selection.

I validated ScreenSift on the LIT-PCBA MAPK1 benchmark using native-ligand redocking, two validated MAPK1 receptor setups, Uni-Dock virtual screening, GNINA rescoring, ligand-similarity scoring, and final candidate ranking.

Why It Exists

Virtual-screening projects mix heterogeneous evidence — chemical similarity, docking scores, and ML rescoring — that are not on the same scale. A single weighted sum can hide useful candidates when one channel is weak or missing. ScreenSift normalizes each channel with its known score direction, assigns interpretable support buckets, and returns the top candidates together with the evidence that ranked them. It ranks and explains candidates; it does not claim experimental activity.

Workflow

The intended end-to-end path is:

dataset + schema + target
  -> molecular audit and canonical ligand table
  -> receptor/box/native-ligand setup
  -> native-ligand redocking validation
  -> optional receptor filtering from redocking pass/fail
  -> ligand 3D/PDBQT preparation
  -> receptor/box-driven Uni-Dock screening
  -> optional GNINA rescoring
  -> ECFP4/Tanimoto similarity evidence
  -> normalized docking/rescoring and similarity evidence
  -> candidate table

A YAML schema maps arbitrary dataset columns (ChEMBL-style or a local CSV) onto ScreenSift fields, and activity labels can be taken directly or derived from thresholds. Curation canonicalizes molecules with RDKit, generates InChIKeys, deduplicates by molecular identity, and records explicit failure reasons for invalid or conflicting rows.

Native Redocking Before Screening

Before the many-ligand screen, ScreenSift can redock each receptor's known native ligand with Uni-Dock to validate the box and docking setup before spending time on library docking. This is report-only by default; with filter_receptors: true it screens only receptor setups that pass the configured RMSD threshold. An optional, fully user-defined auto-tuning step tries a sequence of box/exhaustiveness attempts per receptor and stops at the first that passes — and if filtering is enabled and no receptor passes, the pipeline stops rather than screen against an invalid receptor set.

Evidence, Normalization, And Ranking

Uni-Dock and GNINA produce structure-score columns (unidock_best_score, CNNscore, CNNaffinity, gnina_affinity), auto-detected with the correct score direction; ECFP4/Tanimoto similarity to known actives is computed when no similarity column is supplied. Each score is rescaled independently to 0–1 with its direction handled, combined into one structure_score_norm per ligand, and merged with similarity under a user-chosen policy (max or explicit weighted mean). The find_candidates API and matching CLI expose this selection step directly.

Evidence Buckets

Normalized scores are compared against user-defined cutoffs to label why each compound ranked — without reserving slots for either channel:

ConditionBucket
similarity passes, structure passesconsensus_supported
similarity passes, structure does notanalog_supported

MAPK1 Benchmark Run

The example workflow was run on the LIT-PCBA MAPK1 target using the two validated receptor setups, 4QTA and 6SLG:

StageResult
Dataset audit61,971 curated ligands: 302 active and 61,669 inactive
Receptor setuptwo validated MAPK1 receptors: 4QTA and 6SLG
Score population51,507 ligand rows from 4QTA/6SLG Uni-Dock docking and GNINA rescoring
Similarity evidenceECFP4/Tanimoto similarity to known actives for all 51,507 rows
Complete score rows51,430 complete, 77 failed
Label composition262 known actives, 51,245 known inactives
Ranking policycombined evidence: similarity 0.75 / structure 0.25
Top-100 shortlist6 known actives and 94 known inactives

The scored population contains 262 known actives among 51,507 rows (0.51%), so a random top-100 sample would contain about 0.5 known actives on average. This example shortlist's 6/100 is roughly 11.8× enriched over that random baseline. The active count is the retrospective label composition of the ranked shortlist, not a claim of new MAPK1 inhibitors.

Tests And Validation

The package test suite runs with PYTHONPATH=src python -m pytest tests/; the current local result is 122 tests passing (with 2 upstream MDAnalysis/numpy warnings). Continuous integration runs the suite on GitHub Actions.

Scope

Retrospective, public-data workflow over the LIT-PCBA MAPK1 benchmark. The output is a ranked triage table with explicit evidence channels, treated as separate 2D and 3D evidence rather than a method-vs-method benchmark, and it is not a claim of new MAPK1 inhibitors.

Reproducibility

The standalone repository is a pip-installable package with a runnable MAPK1 example, committed reference outputs to check a run against, tests, and CI. The core find_candidates API installs from PyPI wheels alone; the full docking/prep workflow adds external tools (Meeko, Open Babel, Uni-Dock, GNINA) via the reproducible conda environment.

pip install -e .
python example/mapk1/run_example.py
python -m pytest tests/
